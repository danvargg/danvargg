Case Study
Civil Engineering

Problem

Civil engineers spend hours manually extracting quantities and costs from technical drawings - a process that is slow, expensive, and prone to error.

Solution

A structured extraction pipeline that ingests civil engineering PDFs and automatically generates calculations, material quantities, and full costing reports.

Status

In Progress
Active development

Stack

Python · Claude API · tool_use · Message Batches API · Pydantic

Context

Civil engineering projects generate large volumes of documentation (technical drawings, specifications, and reports) that contain the data needed to estimate material quantities and project costs. Today this is largely a manual process: engineers page through PDFs, extract figures by hand, and enter them into spreadsheets. It is slow, inconsistent across teams, and expensive at the scale of large infrastructure projects.

The solution

The core is a structured extraction pipeline using the Claude API with tool_use, which enforces a JSON schema the model populates field by field. The schema uses required and optional fields, and nullable fields for information that may not appear in every document.

A validation layer catches format mismatches before they propagate downstream. When validation fails, the system sends a follow-up request to Claude that includes the original document, the failed extraction, and the specific error - giving the model the context to self-correct. The retry logic distinguishes between recoverable failures (format issues the model can fix) and unrecoverable ones (information genuinely absent from the source).

Field-level confidence scores route low-confidence extractions to a human review queue, with accuracy tracked by document type and field to surface systematic weak points.

Where It Stands

Currently in the POC phase: gathering representative PDF examples, iterating on the JSON extraction schema, and validating the pipeline against real documents before moving to batch-scale processing.
